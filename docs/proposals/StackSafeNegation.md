# Proposal: Derivation Counters for Stack-Safe Differential Maintenance

## Summary

Differential removal today marks tuples `present ‚Üí unknown` (a 2-bit `TupleState` per row in `include/drlojekyll/Runtime/Table.h`) and re-proves unknown tuples with recursive top-down checker procedures compiled into the generated code. The recursion is unbounded on deep derivation chains and has blown the stack in real use. Worse for the long game, `kUnknown` is control state, not data: correctness depends on ordering ‚Äî mark unknown, let the removal cascade quiesce, only then recheck and replay ‚Äî so the quiesce points are global serialization barriers, and checkers read (and write) shared mutable table state while cascades are in flight.

This proposal replaces the tri-state and the entire top-down checker apparatus with **split per-row derivation counters** (`C_nr`/`C_r`, per Hu, Motik & Horrocks, "Optimised Maintenance of Datalog Materialisations", AAAI 2018 ‚Äî the DRed^c algorithm) and bounded, set-at-a-time, per-stratum OVERDELETE ‚Üí REDERIVE ‚Üí INSERT fixpoints:

- `C_nr` counts a row's currently-firing *nonrecursive* derivations (rule instances whose deriving operator lies outside the row's SCC, plus one for explicit message support); `C_r` counts *recursive* derivations (instances arriving over an inductive back-edge of the same SCC). Presence is `C_nr + C_r > 0`. Counters are **signed** and may transiently dip below zero mid-batch (¬ß5.1.1); at end-of-batch commit both are provably non-negative and asserted so. Nothing replaces `kUnknown` as persistent state.
- Outside recursive SCCs, retraction is exact reference counting: a commutative counter fold whose zero crossing is the propagation trigger. No re-proof of any kind.
- Inside a recursive SCC, a bounded overdelete fixpoint gated on `C_nr ‚â§ 0` runs, followed by rederivation that is **a single counter read** (`C_r > 0` after quiescence ‚áí genuinely surviving support), then the ordinary insert fixpoint. `REDERIVE` is the entire replacement for `BuildTopDownChecker` and its per-node web: no rule evaluation, no scan of predecessors, no recursion, O(|overdeleted|) counter reads.
- Counter updates are atomic RMWs on ‚Ñ§ ‚Äî they commute, and crossing detection reads only the RMW's own before/after snapshot plus one batch-frozen flag, never a mutable flag. Within each phase, updates are therefore applicable out of order and in parallel. The residual serialization is only stratum topological order, two phase boundaries per recursive stratum, fixpoint-round frontiers, and the end-of-batch commit sweep. No per-retraction barriers; no reader whose verdict depends on mid-cascade mutable truth.

Deleted wholesale: `kUnknown` and the tri-state transition methods, every `BuildTopDown*Checker`, the induction recheck/replay machinery (`BuildUnknownRecheck`, `kInductionRechecks`, the codegen re-entry loop), the eager negation coupling (`MaybeRemoveFromNegatedView` / `MaybeReAddToNegatedView` / `PivotAroundNegation`), the publish flusher's checker branch, the PRODUCT insert path's checker double-check, and query-cursor checker filtering. No construct in the design ever recurses; every loop is a vector-driven fixpoint bounded by the rule-instance counts of the old and new materializations.

## 1. Differential removal today

One received message batch drives the entry procedure: additions run the eager bottom-up insertion regions; removals run `InTryMarkUnknown` (`lib/ControlFlow/Build/Build.cpp:1566`) and the `BuildEagerRemovalRegionsImpl` cascade (`Build.cpp:1605`), flipping rows `present ‚Üí unknown` through the same region shapes with dequeue-time polarity dispatch. Unknown rows are later resolved by `ProcedureKind::kTupleFinder` checkers ‚Äî one recursive procedure per (view, column set), generated by `BuildTopDownChecker` (`Build.cpp:408`) and its per-node variants (declared at `Build.h:431‚Äì519`: Induction/Union/Join/Tuple/Negation/Select/Compare/Generator, defined in the per-node files) ‚Äî which recurse across predecessor views to re-prove a row from surviving facts. Consumers of the verdict:

- inductions, via the 4-phase snapshot/output/clear/reseed region and `BuildUnknownRecheck` (`Induction.cpp:599`) over `kInductionRechecks` vectors, plus codegen's outer re-entry loop;
- NEGATE nodes, via the eager two-way coupling (`MaybeRemoveFromNegatedView`/`MaybeReAddToNegatedView`, `Build.cpp:339`, `:365`, whose `PivotAroundNegation` (`Build.cpp:269`) probes the negation-key index via `BuildMaybeScanPartial` (`Build.cpp:312`));
- deletion-capable cross-products, whose insert path double-checks each pairing via `CallTopDownChecker` (`Product.cpp:223`);
- the publish flusher (`Build/Procedure.cpp:290‚Äì335`), which calls a checker on each accumulated output row *of deletion-capable views* (`Procedure.cpp:295`) to decide add-vs-remove publication ‚Äî monotone-backed `@differential` messages accumulate in the same `kMessageOutputs` vectors (created on `message.IsDifferential()` alone, `Procedure.cpp:229‚Äì231`; appended in `Insert.cpp:52‚Äì63`) and are sort-uniqued and published checker-free;
- query cursors, via `ProgramQuery::tuple_checker` (`include/drlojekyll/ControlFlow/Program.h:1120`) and the generated per-row `find_*` filter calls in `lib/CodeGen/CPlusPlus/Database.cpp`.

The checker call graph mirrors the view graph, so a derivation chain of depth *n* costs *n* native stack frames. And because a checker's answer is only meaningful after the removal cascade has quiesced, the whole protocol is a lattice of ordering obligations (snapshot before checker, reseed after output, dequeue-time polarity, checker-as-writer races) hostile to any parallel evaluation strategy.

## 2. The chosen state model, and why

The design space (docs/TODO.md items 2 and 4) is a bookkeeping-vs-reproof spectrum: Backward/Forward keeps zero standing state and pays re-proof (backward-chaining checks against the shared materialization) at deletion time; DBSP/Z-sets keep per-operator integrated weights everywhere (plus per-iteration traces inside fixpoints) and never re-prove anything. Judged with **parallelization as a first-class criterion**, split derivation counters sit at the spectrum's sweet spot: 8 bytes/row of standing state on differential tables only ‚Äî no arrangements beyond the indexes that already exist, no per-(row, iteration) trace multisets ‚Äî yet *zero re-proof*: rederivation inside a recursive SCC collapses to reading `C_r`, and outside SCCs deletion is pure refcounting with no phases at all. Counter updates commute within each phase (an abelian-group fold, exactly the monoid-fold shape TODO item 2 asks for), so the tri-state's per-retraction quiesce barriers dissolve into a static stratum/phase/round lattice; what this model concedes to full Z-sets is DRed-shaped overdeletion churn *confined to recursive SCCs* and pruned by the `C_nr > 0` firewall (a row with any surviving lower-stratum derivation stops the cascade dead). Monotone flows pay nothing. One piece of the Z-set algebra is deliberately borrowed: the batch-boundary delta enumeration (¬ß5.1) is the exact telescoped **signed** delta expansion, which is what makes mixed add/remove batches net correctly.

**Where TODO item 5's reify-into-dataflow trick applies.** Item 2's original sketch ‚Äî "the 'needs re-proof' set is itself a relation" ‚Äî is *dissolved* rather than implemented: there is no re-proof set, because re-proof is a counter read. What does get reified as first-class, relation-shaped data is (a) the per-stratum frontier queues (`delQ`/`addQ` ‚Äî vectors of row ids that crossed zero, exactly the observable/optimizable worklists item 2 wanted the recheck set to become), and (b) the inter-stratum interface, which is a pair of consolidated signed frontiers (`D\A`, `A\D`) per table ‚Äî message-shaped batches. Item 5's request/response transport applies *later*, when phases become distributed: frontier batches ride the message machinery and quiescence detection becomes Dijkstra‚ÄìScholten outstanding-message accounting (¬ß7). Item 5's `@async` functors also compose cleanly with this model: a retracted external response is just a `SubExplicit` on the response relation.

## 3. Data structures

### 3.1 `include/drlojekyll/Runtime/Table.h`

`TupleState`, the `states` array, and all five `TryChange*` methods are deleted. Two table flavors, chosen per data model at codegen time by `QueryView::CanReceiveDeletions()` ‚Äî a compile-time split exactly like today's differential/non-differential checker split, not a runtime dual path:

```cpp
// Monotone relation: presence is "row exists in the log". Insert-only.
template <typename Row>
class Table {
 public:
  struct Insert { bool added; uint32_t id; };  // added == changed == added_row
  Insert TryAdd(const Row &row);               // dedup by value; only mutator
  bool Present(uint32_t id) const;             // always true for stored rows
  // rows / hashes / slots exactly as today; no per-row state array at all.
};

// Differential relation: split derivation counts + per-batch scratch flags.
enum RowFlags : uint8_t {
  kInI      = 1u << 0,  // present at batch start (the frozen "I" of DRed^c)
  kDel      = 1u << 1,  // claimed by the overdeletion set D this batch
  kAdd      = 1u << 2,  // claimed by the addition set A this batch
  kDelNow   = 1u << 3,  // claimed by the CURRENT delete frontier round
  kAddNow   = 1u << 4,  // claimed by the CURRENT insert frontier round
  kExplicit = 1u << 5,  // counted +1 in C_nr for explicit (message) support
};
enum class DerivClass : uint8_t { kNonRecursive, kRecursive };

template <typename Row>
class DiffTable {
 public:
  struct Delta {
    bool crossed;    // this update is a zero-crossing event -> enqueue
    bool added_row;  // fresh log entry -> append id to every index (as today)
    uint32_t id;
  };
  // counts[id]: two SIGNED int32 counters (C_nr, C_r) packed in one 64-bit
  // word, so one atomic RMW returns a consistent (before, after) snapshot
  // of both. Counters may transiently go negative mid-batch ‚Äî a canceling
  // "phantom pair" (¬ß5.1.1) can apply its minus before its plus ‚Äî and are
  // asserted non-negative (per class) at Commit, never per-RMW.
  //
  // Crossing tests read ONLY the RMW snapshot and the batch-frozen kInI
  // bit ‚Äî never kDel/kAdd, which mutate concurrently within a phase:
  Delta AddDerivation(const Row &row, DerivClass c);
      // ++count(c); crossed <=> before_total <= 0 && after_total > 0
  Delta SubDerivation(const Row &row, DerivClass c);
      // --count(c); crossed <=> kInI && after_C_nr <= 0
      // (duplicate enqueues are absorbed by TryClaimDel)
  Delta AddExplicit(const Row &row);  // kExplicit 0->1: ++C_nr, same crossing
                                      // rule as AddDerivation; else no-op
  Delta SubExplicit(const Row &row);  // kExplicit 1->0: --C_nr; else no-op

  // Named membership predicates (single flag-byte reads). These are the
  // ONLY forms in which generated joins may read a differential table;
  // each CHECKMEMBER region names one of them explicitly (¬ß3.3).
  bool InI(uint32_t id) const;         // kInI               ‚Äî batch-start state
  bool InNew(uint32_t id) const;       // (kInI&&!kDel)||kAdd ‚Äî final-so-far
  // Fixpoint-round reads (¬ß5.1); "frontier" = the claim round now draining:
  bool SurvivesSoFar(uint32_t id) const;       // kInI && !kDel
  bool AliveAtClaim(uint32_t id) const;        // kInI && (!kDel || kDelNow)
  bool InNewWithFrontier(uint32_t id) const;   // (kInI&&!kDel)||kAdd
  bool InNewSansFrontier(uint32_t id) const;   // (kInI&&!kDel)||(kAdd&&!kAddNow)
  bool Present(uint32_t id) const;             // post-commit: counts > 0
  bool RecursivelySupported(uint32_t id) const;// C_r > 0 (the REDERIVE test)

  bool TryClaimDel(uint32_t id);  // CAS kDel|kDelNow 0->1 (dequeue dedup);
                                  // debug-asserts C_nr <= 0 at claim
  bool TryClaimAdd(uint32_t id);  // CAS kAdd|kAddNow 0->1
  void RetireDelFrontier(const Vec<uint32_t> &round);  // clear kDelNow bits
  void RetireAddFrontier(const Vec<uint32_t> &round);  // clear kAddNow bits

  // End-of-batch: for each touched row, report 0/1 presence crossings vs
  // kInI (the publish sink consumes them), set kInI := (counts > 0), clear
  // kDel/kAdd; debug-assert C_nr >= 0 && C_r >= 0. Then clear `touched`.
  template <typename Sink> void Commit(Sink &sink);

 private:
  Vec<Row> rows;  Vec<uint64_t> hashes;
  Vec<uint64_t> counts;    // 8 B/row, persistent (two packed int32)
  Vec<uint8_t>  flags;     // 1 B/row storage; batch-transient meaning
  Vec<uint32_t> touched;   // ids whose counts/flags changed this batch
  uint32_t *slots;         // open-addressing set, unchanged
};
```

`Index` is unchanged: append-only, ids added once when `added_row`, liveness filtered at read time by the membership predicates. Cursors remain `(bound params, pos)` and tolerate concurrent appends exactly as today. The run-once CAS discipline of the old `TryChange*` methods is inherited jointly by the atomic packed-counter RMW (each boundary crossing is observed by exactly one update, because the RMW returns a unique before/after snapshot and the crossing predicates depend on nothing mutable) and by `TryClaimDel/Add` at dequeue.

**Flag/count coherence invariant.** After a stratum's phases complete, for every row of its tables: `InNew(id) ‚áî counts > 0`. Sketch: within OVERDELETE counts only decrease and within INSERT only increase (sign-segregated phases); any row ending a phase on the other side of zero from its `kInI` state produced a count-based crossing, was enqueued, and was claimed before phase quiescence (the fixpoint drains its own queue); REDERIVE explicitly enqueues every restored row into `addQ`, so `kDel`-only ‚áí final counts ‚â§ 0 and `kAdd` ‚áí final counts > 0. Downstream strata therefore read coherent state.

### 3.2 Memory cost

Per differential row today: `sizeof(Row)` + 8 (hash) + 1 (state) + ~4.6 (slots at 7/8 load) + 4 per index (`next`). New: the state byte becomes the flags byte (same 1 B) + 8 B counts + an amortized transient `touched` entry per changed row per batch (cleared at commit). Net standing cost: **+8 bytes per row on differential tables; zero on monotone tables**, which lose their state array entirely. Rows are already immortal, so this rides the accepted memory-monotone budget. Counter magnitude overflow at |2¬≥¬π‚àí1| aborts (checked per RMW), the same policy as row-id exhaustion.

### 3.3 Control-flow IR (`include/drlojekyll/ControlFlow/Program.h`)

New region kinds:

- `UPDATECOUNT(table, ¬±, DerivClass)` ‚Äî one counter fold per rule firing, with a body executed only on `crossed` (the shape CHANGETUPLE has today: CAS-like, success body). Emitted inline at join-output sites.
- `CHECKMEMBER(table, MembershipPredicate, present_body, absent_body)` ‚Äî a 2-way (never 3-way) gate naming one of the ¬ß3.1 predicates explicitly (including the negated "key absent in ‚Ä¶" forms used at NEGATE positions). Making the read-state a named IR form (not prose) is deliberate: every join site's frozen-vs-current read is auditable in `-ir-out`.
- `COMMITSWEEP(table)` ‚Äî the end-of-batch commit/publish loop.

Deleted: `CHECKTUPLE`'s 3-way form and `kAbsentOrUnknown`, `CHANGETUPLE`'s tri-state transitions, `ProgramModeSwitchRegion` (polarity becomes *which phase a region sits in*, structural rather than dispatched), `ProcedureKind::kTupleFinder`, `VectorKind::kInductionRechecks`, `ProgramQuery::tuple_checker`.

Per (stratum, differential table): frontier queues `delQ`/`addQ` (row ids that crossed zero ‚Äî drained destructively by the stratum's phases), per-phase claim accumulators `D`/`A`, and per-table consolidated output frontiers `D\A`/`A\D` built at end of stratum (¬ß5.0). All reuse the existing induction input/swap vector machinery (sort-unique preserved; dedup at dequeue via `TryClaim*`).

### 3.4 Negation-column index

The crossover joins of ¬ß5.4 probe the negated view's predecessor keyed on the negation's `input_columns`. The index for this already exists: today's `PivotAroundNegation` requests it through `BuildMaybeScanPartial` (`Build.cpp:312`), which calls `TABLE::GetOrCreateIndex` over the bound key columns and emits an index-backed `TABLESCAN` (a full scan happens only when no columns are bound). The new joins reuse exactly that index request; no new machinery and no performance delta is claimed here.

## 4. Stratification (new dataflow pass)

`lib/DataFlow/Stratify.cpp`, run in `Query::Build` after `TrackDifferentialUpdates`: Tarjan SCC condensation over the view graph ‚Äî column edges, `negated_view` edges, and the INSERT‚ÜíSELECT seams `TrackDifferentialUpdates` already enumerates. Outputs:

- `stratum(view)`: topological SCC order; `stratum(table)` := the stratum of its data model's views (see Open Question 11 for models that straddle strata);
- a `DerivClass` for every table-inserting edge: `kRecursive` iff the deriving view is in the same SCC as the views of the target table's data model, else `kNonRecursive`;
- a clean diagnostic (feature-gap style, like the aggregate/kvindex errors) when a NEGATE's `negated_view` lies inside its own SCC. No stratification check exists today; such programs get order-dependent semantics from the inductive-negation path through `GetOrInitInduction` (`Build/Negate.cpp`). Under this proposal they are rejected loudly. The corpus witness that recursive strata *negating lower strata* remain legal is `tests/OptDiff/cases/disassemble.dr` (`!function` inside the recursive `function_instructions` negates a lower-stratum view).

Only differential strata (any member view `CanReceiveDeletions()`) run the counter machinery; monotone strata run today's eager insertion untouched. A monotone stratum's contribution to downstream differential seeds is its `A\D` frontier ‚Äî simply its rows newly inserted this batch, which the existing insertion vectors already carry ‚Äî and its `D\A` frontier is empty by construction.

## 5. Algorithms

All pseudocode is set-at-a-time; there is no recursion anywhere. Firing enumeration uses the existing TABLEJOIN/index-scan machinery with one side ranging over a frontier vector and the other sides gated by `CHECKMEMBER` predicates.

**Design rule (exactly-once counting):** counter updates happen inline at join-output sites ‚Äî one `Add/SubDerivation` per rule-instance firing (multiset discipline) ‚Äî never buffered through sort-uniqued vectors. Vectors carry only *frontiers* (rows that crossed zero), which may be freely sort-uniqued because `TryClaim*` dedups at dequeue.

### 5.0 Batch skeleton (one received message batch = one epoch)

```
ingest:
  monotone receives: Table::TryAdd + today's eager input vectors (unchanged)
  differential receives: NET each row's explicit adds against its explicit
      removes within the batch, with SET semantics: each side is
      deduplicated and the adds‚à©removes intersection annihilates ‚Äî
      dropped from both sides, presence unchanged (see ¬ß5.5 note);
      then per netted row:
        add:    d := AddExplicit(row);  if d.crossed: addQ[stratum(v)] += d.id
        remove: d := SubExplicit(row);  if d.crossed: delQ[stratum(v)] += d.id
for s in strata, topological order:      # deletion & insertion INTERLEAVED
    if s is differential:                # per stratum (mandatory with negation)
        OVERDELETE(s)                    # drains delQ[s] destructively
        REDERIVE(s)
        INSERT(s)                        # drains addQ[s] destructively
        BUILDFRONTIERS(s)                # consolidated D\A, A\D per table
    else:
        run today's monotone eager insertion for s
for each differential table: Commit(publish_sink)          # ¬ß5.5
for each monotone-backed @differential message:
    sort-unique + publish its kMessageOutputs vector       # unchanged today
```

```
BUILDFRONTIERS(s):
  for each differential table t in s:
      outDel(t) := { id ‚àà D_s(t) : !kAdd(id) }     # D\A ‚Äî net removals
      outAdd(t) := { id ‚àà A_s(t) : !kDel(id) }     # A\D ‚Äî net additions
```

These consolidated signed frontiers are what all downstream (higher-stratum) seed joins range over. A batch with no removals and no negation flips has empty `delQ` everywhere: OVERDELETE and REDERIVE are no-ops and the batch degenerates to the eager insert path ‚Äî the monotone fast path survives intact.

### 5.1 The delta schemas (exactly-once firing enumeration)

Each rule body has a fixed total position order: **lower-strata atoms first** (in stratum order, then syntactic order), **then same-stratum atoms** in syntactic order. Negated atoms always occupy lower-stratum positions (guaranteed by Stratify). There are two enumeration schemas, both instances of the telescoped signed-delta identity `New(B‚ÇÅ‚Ä¶B‚Çô) ‚àí Old(B‚ÇÅ‚Ä¶B‚Çô) = Œ£·µ¢ New^{<i} ‚äó Œî·µ¢ ‚äó Old^{>i}` where `Œî·µ¢ = New·µ¢ ‚àí Old·µ¢` is a **signed** set:

**Seed schema** ‚Äî delta position `i` is a lower-stratum atom; the delta ranges over that stratum's consolidated signed frontier. `‚àí` elements (positive atom in `D\A`; negated atom's key in `A\D`) are enumerated during OVERDELETE and applied as `SubDerivation`; `+` elements (positive atom in `A\D`; negated key in `D\A`) during INSERT as `AddDerivation`. Non-delta reads do not depend on the sign:

| position | read |
|---|---|
| lower `j < i` | `InNew` (negated: key absent in `InNew`) |
| lower `j > i` | `InI` (negated: key absent in `InI`) |
| same-stratum `j` (always `j > i`) | `InI` |

**Fixpoint schema** ‚Äî delta position `i` is a same-stratum atom; the delta ranges over the current claim round (`Œî_D` in OVERDELETE, `Œî_A` in INSERT). All lower positions are `j < i` by the position order:

| position | OVERDELETE (`‚àí`, delta ‚àà Œî_D) | INSERT (`+`, delta ‚àà Œî_A) |
|---|---|---|
| lower `j < i` | `InNew` (negated: absent in `InNew`) | `InNew` (negated: absent in `InNew`) |
| same `j < i` | `SurvivesSoFar` | `InNewWithFrontier` |
| same `j > i` | `AliveAtClaim` | `InNewSansFrontier` |

The `kDelNow`/`kAddNow` frontier bits exist precisely so that an instance whose two same-stratum body atoms are claimed **in the same round** is still counted exactly once: with the earlier position as delta, the later one passes `AliveAtClaim`; with the later as delta, the earlier fails `SurvivesSoFar`. Both schemas are one generated pattern per rule; a missing term is a schema bug visible on every program, not a per-site hand-derivation.

**Net-exactness (case analysis, spot-verified against adversarial traces):** for every rule instance ‚Äî over any combination of {deleted, added, overdeleted, rederived, surviving, genuinely-new} atoms at lower and same-stratum positions ‚Äî the sum of all schema firings is ‚àí1 if the instance is in the old materialization only, +1 if in the new only, and 0 otherwise. Deletions fire with the delta at the *first* changed position (later positions read `InI`-flavored state where the change hasn't happened); additions with the delta at the *last* changed position (earlier positions read `InNew`-flavored state).

#### 5.1.1 Mixed-change instances and phantom pairs

One instance class fires *twice*, with canceling signs: an instance whose earlier-position atom is **added** and later-position atom is **deleted** in the same batch. The `‚àí` term fires (delta = the deleted later atom; the earlier position reads `InNew`, which contains the add) and the `+` term fires (delta = the added earlier atom; the later position reads `InI`, which still contains the deleted row). Neither firing corresponds to an instance of the old or new materialization ‚Äî they are a **phantom pair**, and they always net to zero within the batch, with the `‚àí` landing in OVERDELETE and the `+` in INSERT. This is why counters are signed with commit-time (not per-RMW) non-negativity asserts, and why crossings are count-based:

- Phantom `‚àí` on a never-present head (`kInI` clear): `SubDerivation` takes the count to ‚àí1; the crossing predicate requires `kInI` ‚Üí no crossing, nothing propagates. INSERT's phantom `+` restores ‚àí1 ‚Üí 0; the up-crossing predicate requires `after_total > 0` ‚Üí no crossing. Net silence, correct.
- Phantom `‚àí` on a present head: a genuine down-crossing may fire, overdeleting the row and cascading; the paired `+` in INSERT up-crosses it back, and the standard rederivation-replay path re-adds its consequences. Wasted churn, correct net result, nothing published (Commit compares batch-start `kInI` against final counts).

Worked repair of the transitive-closure trap (`t(X,Y) : edge(X,Y). t(X,Z) : t(X,Y), edge(Y,Z).`; state `edge = {(1,2),(1,4),(4,5)}`, `t(1,5)` supported only by `(t(1,4), edge(4,5))` with `C_r = 1`; batch removes `edge(1,2)`, adds `edge(2,5)`): OVERDELETE overdeletes `t(1,2)`, and the fixpoint's phantom `‚àí` (delta `t(1,2)`, lower position reads `InNew` ‚àã `edge(2,5)`) takes `C_r(t(1,5))` to 0 and overdeletes it; REDERIVE restores neither; INSERT's **seed** `+` (delta `edge(2,5)`, same-stratum position reads `InI` ‚àã `t(1,2)`) restores `C_r(t(1,5))` to 1 and replays; Commit publishes remove `t(1,2)`, add `t(2,5)`, and nothing for `t(1,5)`, whose surviving derivation was never enumerated and whose count is intact.

### 5.2 Deletion ‚Äî `OVERDELETE(s)` then `REDERIVE(s)`

```
OVERDELETE(s):
  D_s := ‚àÖ                                # per-stratum claim accumulator
  N_D := drain(delQ[s])                   # destructive: explicit removals
  # ‚àí seeds per the seed schema: delta over each lower predecessor's D\A
  # frontier (negated positions: over the negated view's A\D frontier):
  for each such stopped firing F:

  for id in D_s:
      if table.RecursivelySupported(id):  # C_r > 0 after quiescence
          addQ[s] += id                   # re-enters via INSERT; ends with
                                          # kDel & kAdd both set => net no
                                          # change, consequences replayed
```

The `C_nr > 0` firewall is the point of the split: a down-crossing requires `C_nr ‚â§ 0`, so a row with any surviving nonrecursive derivation ‚Äî support from a strictly lower, already-final stratum, where cycles are impossible ‚Äî never enters `D`, and the cascade stops at it with no re-proof. A purely cyclically supported row (`C_nr` already 0) enters `Œî_D` on its first recursive decrement ‚Äî the crossing predicate is on the *post-state* `C_nr ‚â§ 0`, not on which counter moved ‚Äî and that is deliberate overdeletion (it may be self-supporting), repaired by REDERIVE. Soundness of REDERIVE (Hu et al. 2018, Thm 1, adapted): after OVERDELETE quiesces, every *old* recursive instance with an atom in `D` or a deleted lower atom has been decremented exactly once, and no `+` has yet been applied, so a surviving `C_r > 0` witnesses at least one derivation from facts that survive the batch. Phantom `‚àí`s can only make `C_r` an *under*-count (they are compensated in INSERT), so REDERIVE may under-restore but never over-restores; under-restored rows are re-added by INSERT's `+` seeds, as in the ¬ß5.1.1 trace.

### 5.4 Negation (antijoin) maintenance

Stratification guarantees `negated_view` sits in a strictly lower, phase-final stratum when the NEGATE's stratum runs, so negated atoms are ordinary lower positions of the ¬ß5.1 schemas with the sign dualized. Three generated pieces replace the entire eager coupling:

- **Forward pass** (a positive delta arrives at the NEGATE): gate on `!negated_table.InNew(key)` ‚Äî one `CHECKMEMBER` read, exactly the "lower `j<i`, negated" schema row. No checker, no third state.
- **Negated view gained rows** (its `A\D` frontier): these are `‚àí` deltas ‚Äî each match against the NEGATE's predecessor via the ¬ß3.4 index (other positions per the seed schema) is a stopped instance ‚áí `SubDerivation(head, class)`, seeding this stratum's OVERDELETE.
- **Negated view lost rows** (its `D\A` frontier): `+` deltas ‚áí `AddDerivation(head, class)`, seeding INSERT.

This crossover ‚Äî an addition to the negated view is a deletion of `not B` and vice versa (Motik et al. 2019, Ex. 18) ‚Äî is why deletion and insertion interleave per stratum instead of running as two global passes. Unit relations (desugared conditions) need no special case: the token row `(true)` has counters like any row, and a condition flip is a crossing that feeds these same joins. Same-batch flip-flops of a negated row (add-then-remove) are netted at ingest for explicit rows and net to `kDel‚àßkAdd` (excluded from both frontiers, coherent counts) for derived rows, so no phantom presence ever reaches these joins. `@never` keeps its monotone contract: the negated side is a monotone `Table`, the gate is `Present`, no deltas ever.

### 5.5 Consistent reads: commit, publish, queries

`Commit` is the only externally visible barrier, and it coincides with the existing end-of-batch promise:

```
Commit(sink):
  debug-assert delQ/addQ empty for every stratum
  for id in touched (deduplicated):
      debug-assert C_nr(id) >= 0 && C_r(id) >= 0
      debug-assert (flags[id] & kDelNow) == 0 && (flags[id] & kAddNow) == 0
      was := flags[id] & kInI;  now := (counts[id] > 0)
      if was != now and this table backs a @differential transmit view:
          sink.publish(RowAt(id), /*added=*/now)
      set kInI := now; clear kDel|kAdd
  clear touched
```

- **Publish (differential tables)**: a row whose batch-start presence differs from its final presence emits exactly one `log.msg(args, added)`. Delete-then-rederive publishes nothing. The flusher's checker branch (`Procedure.cpp:295`) is deleted.
- **Publish (monotone flows feeding `@differential` messages)**: unchanged ‚Äî they keep today's `kMessageOutputs` accumulation, sort-unique, and checker-free publish (`Procedure.cpp:229‚Äì231`, `Insert.cpp:52‚Äì63`). Only differential-table-backed messages move to the commit sweep, which changes their publication *order* relative to today's flusher ‚Äî a golden-churn risk audited in Stage 0 (the sweep can sort its touched set before publishing if order determinism is needed).
- **Explicit add/remove of the same row in one batch** annihilates at ingest (¬ß5.0): each vector is deduplicated and the adds‚à©removes intersection is dropped from both ‚Äî the unordered pair is *defined* as leaving presence unchanged. A deterministic contract replacing today's dequeue-order-dependent outcome; assert/retract are idempotent, so duplicate ops within a batch are meaningless and excess retractions can never outvote an add. Signed off in Open Question 3.
- **Query cursors** run between batches (unchanged contract) and filter each index-chain candidate with `Present(id)` ‚Äî one 8-byte load replacing a recursive `find_*` call. `ProgramQuery::tuple_checker` and the codegen checker filtering are deleted.
- **Forcing functions** (`kQueryMessageInjector`) run a complete mini-epoch ‚Äî ingest, strata in order, commit ‚Äî before the cursor scan, so no query observes pre-commit state.

### 5.6 Induction / recursion

A recursive SCC *is* a differential stratum; its INDUCTION region becomes the OVERDELETE ‚Üí REDERIVE ‚Üí INSERT SERIES with the two fixpoint loops using the existing induction vector skeleton. One D‚ÜíR‚ÜíI pass per SCC per epoch suffices: after stratification no negation lives inside the SCC, so INSERT cannot generate deletions within the stratum. Deleted accordingly: the shared add/remove vector with dequeue-time polarity dispatch, the 4-phase snapshot/output/clear/reseed output region, `BuildUnknownRecheck`, and codegen's outer `for (bool reenter...)` re-entry loop. Nested SCCs are consecutive strata; "the inner fixpoint completes first" is now just topological stratum order.

## 7. Parallelization

**What commutes, precisely.** Every `AddDerivation`/`SubDerivation` is one atomic RMW on the packed signed counter pair; addition on ‚Ñ§ commutes, so **within a phase** ‚Äî where all updates share one sign ‚Äî firings may be enumerated by any number of workers in any order: shard the frontier vector, run the delta joins data-parallel, fold counters. The crossing decision reads only the RMW's returned before/after snapshot plus the batch-frozen `kInI` bit; it never reads `kDel`/`kAdd`, so no stale-flag interleaving can suppress or duplicate a crossing. Exactly one update observes each boundary crossing; duplicate frontier entries are absorbed by the `TryClaim*` CAS at dequeue. Membership predicates are stable within a phase in the direction that matters: `kInI` is frozen for the whole batch, and `kDel`/`kAdd` grow monotonically within their phases. Mixed-sign concurrency is **not** claimed: the D‚ÜíR‚ÜíI phase boundaries are load-bearing (they are what lets a plain counter read serve as REDERIVE), and they appear in the serialization list below.

**Ordering-independence example (within-phase).** Program: `p(X) : a(X). p(X) : b(X). p(X) : c(X). q(X) : p(X).`, all differential. State: `p(1)` has `C_nr = 2` (via `a(1)`, `b(1)`), `q(1)` has `C_nr = 1`. One batch removes `a(1)`, `b(1)` and adds `c(1)`. In OVERDELETE, workers W1 and W2 race the two `SubDerivation(p(1))` folds: under either interleaving one RMW returns (2‚Üí1, post-`C_nr` > 0, no crossing) and the other (1‚Üí0, crossing) ‚Äî exactly one worker enqueues `p(1)`; the cascade then overdeletes `q(1)`. The `c(1)` add runs in INSERT (phase boundary, not a race): `AddDerivation(p(1))` 0‚Üí1 up-crosses, `p(1)` is claimed `kAdd`, and the fixpoint replays `q(1)`. Commit sees `kInI` = final presence for both rows and publishes nothing ‚Äî the same final database under every within-phase interleaving. Under the tri-state, this same shape requires a full cascade-quiesce-recheck cycle with checker write-backs.

**Residual serialization frontier**, all structural, none per-retraction:

1. **Stratum topological order** ‚Äî semantic: negation must read a settled lower stratum.
2. **Two phase boundaries per differential stratum** ‚Äî OVERDELETE must quiesce before REDERIVE reads `C_r`; REDERIVE before INSERT. These also sign-segregate the counter folds, which is what keeps crossings count-decidable. Detected by frontier-vector emptiness (quiescence *detection*, not a lock).
3. **Fixpoint-round frontiers** inside ¬ß5.2/¬ß5.3 ‚Äî kept for the simple read-only/insert-only vector discipline; removable later RDFox-style.
4. **The end-of-batch commit sweep** ‚Äî the already-promised output atomicity.

When workers or distribution arrive, frontier batches are already message-shaped (¬ß2); phase-quiescence across a distributed stratum becomes Dijkstra‚ÄìScholten outstanding-request accounting on those batches (epoch isolation of `kInI` across distributed batches is Open Question 10). The sequential build remains the correct one-worker degenerate case, so parallel execution is a runtime-scheduler change, not a semantics change.

## 8. Correctness argument

- **Stratified negation stays sound.** Strata process in topological order with deletion and insertion interleaved per stratum; lower strata are phase-final (their `D\A`/`A\D` frontiers built) before any NEGATE reads them, and the gains/losses of a negated view cross into consumer strata only through the two crossover joins. The Stratify pass supplies, as an enforced invariant, the precondition the current codebase never checks.
- **Net-exact counting.** By the ¬ß5.1 case analysis, each rule instance of the old materialization nets ‚àí1, each instance of the new nets +1, and everything else nets 0 ‚Äî including phantom pairs, which cancel within the batch with the `‚àí` in OVERDELETE and the `+` in INSERT. Final counters therefore equal from-scratch derivation counts, which is why they are provably non-negative at Commit (asserted per class) even though they may transiently dip below zero mid-batch.
- **Termination; bounded work.** Each old instance is decremented at most once and each new instance incremented at most once, plus one canceling pair per mixed-change instance, so OVERDELETE + INSERT perform at most |inst(old)| + |inst(new)| + 2¬∑|inst(mixed)| folds ‚Äî the DRed worst-case bound, with the `C_nr` firewall pruning far below it in practice. Every loop is a vector-driven round; static call depth is 1. A derivation chain of depth 10‚Åµ whose base is retracted costs 10‚Åµ *rounds of a loop*, O(1) stack.
- **No lost tuples.** A row ending the batch with `counts ‚â§ 0` lost every counted derivation (net-exactness). A row overdeleted with surviving recursive support is restored by REDERIVE (`C_r > 0` post-quiescence is a sound witness, ¬ß5.2); a row whose surviving support was masked by a phantom `‚àí` is restored by the phantom's paired `+` in INSERT (¬ß5.1.1), and in both cases its consequences are replayed by the insert fixpoint.
- **No spurious tuples.** Cyclic self-support cannot retain a row: a pure cycle's rows all have `C_nr = 0`, the overdelete wave traverses the cycle decrementing every in-cycle instance exactly once, and `C_r` reaches 0 exactly when no external support remains ‚Äî the case single-counter counting gets wrong and the nr/r split exists to fix. Phantom `+`s cannot mint presence: an `AddDerivation` up-crosses only when the final-so-far count is genuinely positive, and net-exactness guarantees a positive commit count is backed by real instances. The fragile `kUnknown ‚Üí kAbsent` "can't be its own base case" protocol is replaced by a structural property.
- **FINDINGS ledger, by construction.** *F11b* (insert-side joins skipping transiently-unknown siblings; lost derivations): impossible ‚Äî there is no transient third state; INSERT joins read `InNew*` predicates that only grow during the phase. *F11a* (checkers bottoming out on unmaterialized sibling support): support is recorded as a count on the head's own table at derivation time; nothing ever consults predecessors to re-prove. *F15* (self-defeating checker gate on a just-flipped condition): no checkers exist to gate anything; a condition flip is a counter crossing feeding the standard crossover joins. *F12* (missing hand-derived delta term): the seed and fixpoint schemas are one generated pattern per rule, not N hand-derived sites. *F8*-class (condition semantics lost in checker wrap): the checker web no longer exists; unit relations use the uniform ¬ß5.4 mechanism.
- **The new bug class this design introduces: counter drift.** A double- or missed count is silent until a zero crossing and ‚Äî like F11/F15 ‚Äî is invisible to golden diffing if all modes share it. The adversarial review of an earlier draft of this document found exactly such defects in its read-state table (mis-netted mixed add/remove batches); the ¬ß5.1 schemas are the repaired, re-traced version, and the traces are kept as regression cases (Stage 0). Mitigations are part of the design: (a) commit-time non-negativity asserts per counter class, plus per-RMW magnitude-overflow asserts and per-claim `C_nr ‚â§ 0` asserts; (b) `DiffTable::DebugValidateCounts()`, which recomputes all counters from a from-scratch semi-naive evaluation and compares, run at the end of every OptDiff driver in debug builds; (c) debug asserts that all frontier queues are empty at commit and that monotone tables never receive a `SubDerivation`; (d) the Stage 2 reference oracle below, which runs as an independent execution variant on every differential case.

## 10. Alternatives considered

Three competing designs were developed to the same depth and judged by a three-member panel (parallelism, correctness, integration lenses). This proposal is the panel's aggregate winner; the others lost as follows.

**Z-set weighted evaluation (DBSP pole).** Signed per-row weights with iteration-indexed traces inside inductive SCCs; retraction is a ‚àí1 delta through the same operators, delta rules derived mechanically from the bilinear join rule. The best endgame: full commutativity with no phase barriers at all, the `@product` and F11a table-diet wins fall out, and it is the principled base for aggregates. It lost because its central novel mechanism ‚Äî the per-round trace replay for recursive SCCs ‚Äî was shown unsound as written (re-crossing visibility events were folded back in as weight corrections, letting a self-supported row survive deletion of its base), because it carries the largest standing state (+3 B/row plus ~12 B per (row, round) in inductions, with trace GC unspecified), and because versioned in-SCC reads (`PresentAtRound`) have no analogue in today's runtime ‚Äî the largest rewrite for the least-proven core. Its signed-delta batch algebra, however, was vindicated by the adversarial review of this proposal's earlier draft and is now incorporated at the batch boundary (¬ß5.1); the door stays open for a stratum-local move to full weights (e.g. for aggregates) without undoing this design.

**Reified dual-program DRed (epoch-phased overdelete/rederive/insert with mask bits).** The cleanest pure-integration fit: mask bits in the existing state byte (zero persistent memory delta), dual regions in the existing vocabulary, and the best-understood theory (DRed/Motik Alg. 4). It lost on the primary criterion: by its own accounting it *relocates* the tri-state into three phase barriers per stratum rather than dissolving it ‚Äî retraction remains ordered control, not a commutative fold ‚Äî and it has no overdeletion firewall, so the D-cone (and the serialized round count) is the largest of the field, paid even when all support survives. Its own escape hatch for that churn is to bolt on precisely this proposal's counters later ‚Äî two migrations to reach the same end state. Its judged defect (per-stratum sealing made "old state" reads of already-sealed lower strata return new state, silently retaining unsupported tuples) is also instructive: correctness of any phase design hinges on a batch-frozen old-state read, which this proposal gets natively from `kInI`. Grafted from it: the named-read-predicate discipline (¬ß3.3), the Dijkstra‚ÄìScholten framing (¬ß7), and the honest golden-churn contingency (Stage 0/3).

**Set-at-a-time Backward/Forward (FBF).** The minimum-bookkeeping pole: transient mark bits only, with backward proof search recast as demand rounds plus a forward saturation fixpoint. It fixes the stack and its frozen-snapshot/monotone-mark discipline is genuinely elegant, but it keeps re-proof itself: check/saturate verdict rounds are extra barriers *nested inside* each delete round, scaling with proof depth √ó cascade depth ‚Äî the set-at-a-time reification of exactly the check-before-propagate ordering this effort exists to dissolve, as TODO item 2 already anticipated ("B/F inherits much of the ordering problem even though it fixes the stack"). It also carries the largest mechanism count (seven interacting mark bits, five phase families, a budget dial) and a judged lost-tuple defect in its negation predicates. Grafted from it: the deep-chain retraction sentinel, the forcing-function-as-mini-epoch rule (¬ß5.5), and O(touched) transient clearing.

## 11. Open questions

1. **In-SCC negation semantics.** The Stratify diagnostic rejects programs whose NEGATE's `negated_view` shares its SCC ‚Äî programs the current inductive-negation path accepts with order-dependent semantics. If the Stage 1 corpus audit finds real usage, the options are: keep the diagnostic (recommended ‚Äî matching DDlog/DBSP/Souffl√©), or adopt Motik-style per-iteration trace counters for exactly those SCCs (heavier bookkeeping, confined). Needs an owner decision if the audit hits.
2. **Publish semantics: flaps and order.** Whether any golden depends on the flusher publishing transient delete-then-rederive flaps, or on its publication order for differential-table-backed messages (Stage 0 audit). If so: one reviewed `--bless` with the oracle as referee (sorting the commit sweep's output is available for order determinism), or ‚Äî if flap emission turns out to be a contract someone relies on ‚Äî a per-message option, which would complicate the commit sweep. Default position: net-change publication is the correct contract.
3. **Same-batch explicit add+remove contract ‚Äî DECIDED (owner sign-off, 2026-07-11, incl. the present-row boundary case): set semantics with annihilation.** Per batch, each of a message's add/remove vectors is deduplicated and the adds‚à©removes intersection is dropped from both sides. The owner's framing, which is the normative statement: *the intersection has NO effect on IDB state through the explicit channel ‚Äî a row asserted and retracted in one batch keeps exactly the presence the rest of the program can prove for it.* The explicit bit is one derivation among many; an annihilated pair neither adds nor subtracts support, so the row stays present iff surviving explicit support from earlier epochs or an alternate rule derivation holds it up (`Present = C_nr + C_r > 0` counts those independently). Rationale: within a batch the two vectors carry no relative order, so any "last writer" is undefined; annihilation is the unique order-free resolution consistent with idempotent assert/retract, and deduplication forecloses the arithmetic-netting hazard where duplicated retractions outvote an add ({+x, ‚àíx, ‚àíx} is a no-op, not a removal). Rejected alternatives, recorded: arithmetic sign-of-net (multiplicity-sensitive), add-wins/remove-wins bias (changes flip-flap observables; remove-wins would re-canonize the old dequeue-order accident), ordered-op-log LWW (the only order-faithful contract, but an API change; producers who need intra-batch order can split batches ‚Äî order is honored at epoch granularity). Implemented in `NetBatch` (Runtime/Vec.h) and mirrored at the oracle's two netting sites. Relatedly, `kExplicit` stays one set-semantics bit; if bag-semantics EDB ever becomes desirable, it generalizes to a third counter ‚Äî not designed here.
4. **Fixpoint-round barriers.** The round structure buys the simple read-only/insert-only vector discipline; RDFox shows rounds are removable under a linearizable insertion order. Keep rounds until profiling of Stage 6 says otherwise.
5. **Counter width.** Signed `int32_t` pairs packed in 64 bits; dense multi-way joins could overflow. Policy: saturate-and-abort like row-id exhaustion, with an unpacked 2√ó64-bit per-table escape if a real program hits it.
6. **Commit-sweep dedup.** `touched` can hold duplicates for hot rows; a per-row touched-this-batch flag bit (dedup at append) trades a branch for vector size. 7. **Overdeletion and phantom churn.** Two churn sources, both net-exact but not surgical: (a) in pure cycles (`C_nr = 0` throughout, e.g. cycle-heavy transitive closure) the overdelete wave can traverse the whole SCC cone before REDERIVE restores it; (b) mixed add/remove batches pay a canceled ‚àí/+ pair ‚Äî with full overdelete-and-replay of consequences when the phantom `‚àí` down-crosses a present row (¬ß5.1.1) ‚Äî proportional to the number of mixed-change instances. Both are bounded by old+new instance counts; if a workload hits either hard, the stratum-local weight escape (¬ß10) is the remedy. **[FIRST NUMBERS at the bench-harness epoch, 2026-07-14 (bench/BASELINE.md Q2): (a) quantified ‚Äî a planted-ring remove batch does ~63x the net change in fold work, 815x slower than naive recompute; pure cycles are the engine's worst regime by three orders of magnitude. On mixed-support graphs the C_nr firewall holds: retraction/insertion ~1.13 at equal fan-out. (b) observed live at scale ‚Äî the F17 stale-drop gates fire on 94/100 mixed phantom batches (del side).]**
8. **Functor re-execution on delete paths.** Stopped-firing enumeration re-runs MAP functors on deleted inputs ‚Äî pure/deterministic functors only, the restriction that already holds. `@range(*)` functors with large outputs pay the re-invocation on retraction; memoizing MAP outputs in a table is the standard fix if it matters.
9. **Inline multiset counting vs. vector plumbing.** The exactly-once discipline requires counter folds at join-output sites, never routed through sort-uniqued vectors ‚Äî a restructuring of the current set-oriented propagation plumbing. The seed/fixpoint schemas concentrate correctness in one generated pattern (good) but make that pattern the single point of failure ‚Äî an earlier draft's version of it was broken by adversarial tracing on mixed batches; the Stage 0 sentinels, Stage 2 oracle, and the debug recount exist for exactly this reason.
10. **Distributed epoch isolation (Stage 6).** `kInI` is defined per batch; the sequential and shared-memory-parallel designs freeze it trivially, but a distributed stratum has no specified mechanism for isolating one epoch's frozen `I` from the next while frontier batches are in flight. Unresolved; Stage 6 cannot start without a design here.
11. **Tables whose data-model views span multiple strata.** If data-model merging places one TABLE's inserting views in different SCCs, which stratum's D/R/I owns the table's phases is unspecified. `TryClaim*` dedup and the frozen `kInI` prevented constructing a wrong outcome in review, but that is absence of a counterexample, not a spec. Stage 1 must audit whether the corpus produces such models and either forbid the shape in data-model merging or assign ownership (candidate rule: the highest stratum among the model's views, with lower views' insertions treated as seeds).
12. **Aggregates and KV indices** remain feature gaps: non-monotone aggregates (MIN/MAX) need retraction-aware per-key state that counters do not provide. Linear aggregates (COUNT/SUM) become natural in a future weight extension (¬ß10).
