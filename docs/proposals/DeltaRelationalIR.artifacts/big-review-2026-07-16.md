# Delta-relational-IR epoch — independent big review (at `delta-relational-ir` @ 8c1d169)

**Note on setup:** my assigned worktree was cut from the wrong ref (354e772, pre-epoch); I detached it to 8c1d169, the epoch tip, and reviewed that. Everything below cites that tree.

## Verdict

The epoch's landed state is **sound as shipped**: every emission cutover (families #1–#3) was gated raw-byte-identical against the pre-R2 emitter, the DeriveDRStrata port is a faithful line-by-line reproduction of the deleted integer lift, RQ5b's color refinement is genuinely conservative against every `Equals` implementation, and the differential invariants (seed-before-drain, E-17 deferral, B-7 in-round clears, one-arm-pair, monotone-side arms, context-keyed negate gates) are all preserved in the lowering paths I walked. I found **no live miscompile**.

But the ledger oversells the *go-forward* safety architecture. The central claim — "the F17/F18/F22 bug class dies when the E-16 shared-state web becomes checked data" — is realized only for **inventory and structure**, not for **predicates**. The lowering consumes none of the DR ops' predicate payload: `claim_gate`, `gate_pred`, the plan spines, per-arm effects, `pinned_order`, `dep_edges`, and `DRRound::body_ops/output_ops` are all validator-only or entirely unread. The F17 gate lives in CLAIM codegen, the F18 context ternary in `EmitChainStep`, the exactly-once matrix in `EmitJoinFire` — each now *duplicated* in the DR model with no cross-check tying model to emitter. Meanwhile the family-#3 retirements left the remaining validators materially weaker than the ledger implies (details below). The real safety net today is behavioral: 159 stdout goldens ×4 modes + the oracle sidecars — a good net, with named blind spots.

**Before R3c-ii:** (1) fix the R3c staging contradiction (F14 retirement flips `aggregate_1`/`kvindex_1–4` to compiling while their goldens only exist after R3d — the suite cannot be green in between; land R3c-ii+R3d as one suite-gated unit or re-stage); (2) fix StateCell's `Touched()` API (unsound mutable ref); (3) decide explicitly whether lowering will consume DR predicate data (recommended) or record the deferral. **Before merge to main:** write the missing R3a/R3b ledger records with the StateCell deviations named, close the two dangling "for ratification" flags (B-2 amendment, B-14 restatement), and do the CLAUDE.md sweep (untouched all epoch — stale suite count, stale expected-diagnostic list).

## Findings, most severe first

**1. [ARCH] The DR-IR's predicate payload is decorative; predicate authority stayed in the Emit\* templates, duplicated with no cross-check.**
Grep-verified: nothing in `lib/ControlFlow/Build/` outside DR.cpp reads `pinned_order`, `dep_edges`, `arms`/PlanNode spines, `claim_gate`, `gate_pred`, `arm_reads`, or `round.body_ops/output_ops`. `LowerDRFlow`/`LowerDRRounds` (Stratum.cpp:1213, 1520) use DR ops only as *keys* (which table/join/negate, which stratum, which sign) and re-derive all column/predicate machinery from the Query via the surviving `EmitChainStep`/`EmitJoinFire`/`EmitClaimDrain`. Consequence: the claim-relative matrix exists twice — `FixpointSamePred` (DR.cpp:472-477, validator model) vs the real one (Stratum.cpp:793-800); the negate-gate rule twice — `NegateGatePred` (DR.cpp:346-354) vs `in_fixpoint ? kInNew : kInI` (Stratum.cpp:335). If someone reintroduces the F18 sign-keyed read in `EmitChainStep`, **every DR validator stays green**; only goldens/oracle catch it, exactly as before the epoch. Recommendation: as an R2 family #4-ish diff, thread the DR op's pred into `BuildCheckMember`/`EmitJoinFire`'s scan list (or at minimum add an always-on cross-check that the emitter's chosen predicate equals the DR op's) — that is the diff that actually kills the bug class.

**2. [LATENT] `claim_gate` cannot express the F17 gate it claims to carry.**
`DROp::claim_gate` is `Pred::kNetDeleted`/`kNetAdded` (DR.cpp:1153; DR.h:428 admits "kNetDeleted-*flavored* C_nr<=0"). The actual F17 gates are `TryClaimDel: C_nr ≤ 0` / `TryClaimAdd: total > 0` — semantically different predicates from `kNetDeleted` (= kDel&&!kAdd) / `kNetAdded` (= kAdd&&!kDel&&!kInI, the E-12 lesson). V-CLAIM-GATE (DR.cpp:1963-1969) checks the constructor stored the enum the constructor stores — self-satisfied. Harmless while unconsumed; **wrong-by-construction if any future lowering emits the gate from this data**. Fix the encoding (its own gate enum, or the actual counter predicate) before anything consumes it. The B-3 promise ("mandatory data … never a template flourish") is currently half-true: carried, trivially asserted, unconsumed.

**3. [VALIDATOR WEAKNESS] Three checks are weaker than documented; one actively erases evidence.**
- `emit_hazard` (DR.cpp:2684-2690): a non-carried hazard whose natural direction runs against the band key is **silently flipped** to follow the key. A genuine construction bug producing a backward dependence is legitimized, not reported. The ledger's R1d record admits V-LINEAR/V-ORDER-CONSISTENT are "self-satisfied by construction," but the flip is stronger than self-satisfaction — it destroys the counter-evidence.
- **V-LOOP doc-vs-code gap** (DR.cpp:2891-2911): the comment says "we assert the loop-carried RAW has a matching intra-scope WAR (its drain-before-refill witness)"; the code checks only `kind ∈ {RAW, WAR}` and index range. As written V-LOOP essentially cannot fail. The ledger lists V-LOOP among the "load-bearing checks that CAN fail on a perturbation" — I disagree; only V-RETIRE-AFTER and V-READY genuinely can.
- **RAW/WAR classification keyed on construction order**, not semantics (the pair loops at DR.cpp:2701-2745 take `x = accs[i], i<j` in op-construction order), so some true RAW pairs are recorded WAR and skip V-READY.
Recommendation: implement the documented V-LOOP witness check; make `emit_hazard` abort (or at least record) on a non-carried backward hazard instead of flipping.

**4. [DELETION AUDIT] Post-family-#3, the census is no longer independent, and the branch-multiset equality is argument-only.**
`ValidateDROps`' "V-OLD-EQUIV census" (DR.cpp:2122-2256) recomputes expected branches by **calling the same `SuffixesOf`** and the same helper predicates construction used — a shared-helper bug is invisible to it. The deleted `V-OLD-EQUIV(branch-MULTISET)` leg + `SeedDRStrata` pairing were the live proof that the memoized suffix worklist equals the old per-path DFS; that equality (load-bearing for `negated_tables_ready` and hence strata) now rests solely on the design argument at DR.cpp:136-147 plus `reconverge_1`. The sub-audit confirmed the argument is sound (suffix- not outcome-memoization preserves per-path multiplicity; both old and new share the DAG precondition) — **EQUIVALENT-WITH-CAVEATS**. Answer to "what now catches a plausible-but-wrong inventory": only the behavioral net (goldens + oracle). It is adequate for current shapes; named blind spots: induction-owned-negate-in-SCC (the family-#3 review-fix #2 shape, corpus-unexercised — the fix's phase-ownership filter at Stratum.cpp:1754-1757 must stay in lockstep with the init predicate at DR.cpp:1675, with no assert tying them), and table-less plumbing deeper than ~2 hops.

**5. [DeriveDRStrata] Faithful port; one shared wart preserved.**
Rule-by-rule equivalent to the deleted lift (init, `ready_after/ready_across/negated_tables_ready`, all five propagation bodies incl. the deliberate product-strict/crossover-exempt asymmetry, SCC pinning, join-terminal reassignment); order-independent monotone fixpoint, so map-iteration nondeterminism is benign. The `operator[]` pollution of `drain_stratum` by non-phase fold targets (DR.cpp:1802/1819) existed identically in the old code; the consumer-side filter reproduces old behavior. B-14-style byte-identity on the anchors genuinely pins the integers (strata determine region nesting). Verdict: **equivalent for every accepted shape**.

**6. [RQ5b] CONSERVATIVE — verdict stands, one latent coupling to pin.**
Full re-audit found no shape that splits an Equals-equal pair. The sharpest near-miss: HashInit folds deletion flags that Join/Merge/Map/KVIndex/Insert/Negate `Equals` don't compare — safe only because `TrackDifferentialUpdates` derives the flags structurally (Equals-equal ⇒ equal flags). Second latent coupling: the generic input/attached-column folds are no-ops on JOIN/SELECT/MERGE **only because those lists happen to be empty** (asserted for JOIN at Join.cpp:41, incidental elsewhere) — a future pass attaching columns there would silently start splitting pairs. Recommendation: one assert/comment at the folds in Optimize.cpp making that invariant defended. Refinement convergence, MERGE positional order, decl-Id direction, group_ids exclusion: all clean.

**7. [StateCell] Substantially sound; one unsound API, several unrecorded deviations, key contracts untested.**
The two-word split, one-net-pair (`new != old` matching the commit sweep's `was != now`), touched-dedup, and allocator-seam discipline are correct; the @recompute POD handles are stable *because* pools hold `Vec*` and the store is (implicitly) non-movable — an undocumented dependency. Must-fix pre-R3c-ii: **`Touched()` returns a mutable ref and sorts on read** (StateCell.h:324-327; spec says `const`) — a caller can desynchronize `touched` from `touched_flag` (double-append / lost-touch), and R3c's `emit_touched` lowering will consume exactly this API. Record-as-residue: unbounded within-group @recompute membership (violates the §1.3-III O(group-size) cost claim); no overflow/below-zero assert in `Recompute::Fold` (StateCell.h:167); implicit (not explicit) move deletion. Untested: handle survival across rehash under @recompute, teardown under a real arena, group-exhaustion abort, mid-epoch `DebugValidate` precondition.

**9. [R3c DIRECTION] Sequencing is right in outline; three specific hazards.**
The R3a→R3b→R3c-i(pragmas)→R3c-ii→R3d ladder is correctly ordered (corpus annotation needs the pragma surface; F14 retirement in the same diff as lowering honors F-2). Hazards: (i) **the golden gap** — R3c-ii removes the runall.sh expected-diagnostic encoding for `aggregate_1`/`kvindex_1–4` (and kvindex_1's by-design mode split disappears, unaddressed by the spec) but their goldens are only blessed from oracle truth in R3d; the suite is red in between, contradicting the every-diff-suite-PASS rule — land R3c-ii+R3d as one gated unit or keep the diag encoding until R3d and record the F-2 deviation.
