(3) E-17 SCC deferral — RAW on kAdd from every INSERT writer → the − FRONTIER_FILTER, forcing it into INSERT.output (§4.3; both IRs host all SCC filters in INSERT output ir:265-290 / 367-400); negative space via V-DEFER. (5) round vector-swap discipline — clear+reappend round vecs excused from RAW, re-exported by FIXPOINT_ROUND (§1, §4.6); tc ir:129-131 clears, d5rn ir:211-214. (6) sort-unique-before-drain — per-vec unique-contract on the def→drain edge, asserted at drain placement (§1/G6; tc's 10 sites match vector-unique lines, d5rn matches). BROKEN derivations: E-17-adjacent FEEDBACK edge is a plain RAW that makes the dep graph CYCLIC so no topo sort exists (finding 1, HIGH); COMMIT-band-last table-id order has no forcing edge so V-LINEAR cannot certify it (finding 2, HIGH). NO extra prose attributes are smuggled for the working six — they ARE derived from effect intersection — but the two broken ones and the multi-arm precision loss mean the '§4 ordering-is-dependence' claim is NOT yet complete. Negative-space detection: present and adequate for the six working orderings; ABSENT for feedback (V-LINEAR wrongly rejects the valid program) and toothless for commit order (V-LINEAR accepts all 24 permutations of the sweeps).

## [HIGH] The single-pass frontier-feedback edge (d5rn) is a RAW that points BACKWARD against the pinned order, so §4.6's 'PinnedOrder is a topological sort of DepGraph' and V-LINEAR's 'no op precedes a producer it reads' are literally violated. The vocabulary derives this ordering only by ASSERTING (§4.5) that 'the pinned linearization must (and does) respect' the feedback RAW — but a plain RAW here creates a graph CYCLE for which no topological sort exists.

EVIDENCE: Verified in d5_recursive_negate.program.ir: $net_removals:59 and $net_additions:62 are DRAINED by PIVOT_ASSEMBLE at ir:166 and ir:169 (seed band, textually early), but PRODUCED (vector-append) by the deferred FRONTIER_FILTERs at ir:372 (T4 reach, -) and ir:388 (T4 reach, +) inside the INSERT induction `output` region (textually late). Per §4 'append=write, drain=read' ⇒ RAW is producer→consumer = ir:372→ir:166 and ir:388→ir:169, i.e. a late op → an early op. Combined with the forward stratum-order RAWs (seed → later phases) this is a CYCLE. Spec §4.5 (v3) claims this is 'an intra-flow producer→consumer edge the pinned linearization must (and does) respect' and 'makes the single-pass feedback a graph fact, not implicit ordering (d5rn gap 1)', but a topological-sort-checked linearization CANNOT respect a cyclic dep set. d5rn.drir.md §gap-1 (line 359) originally flagged exactly this as the vocabulary hole; v3 §4.5 names it but does not resolve the cyclicity.

RECOMMENDATION: The feedback edge is NOT a same-band RAW; it is a LOOP-CARRIED dependence (next-epoch consumer reads previous-epoch producer, empty on first pass — d5rn.drir.md:99 'on the FIRST seed pass they are empty'). v3 must classify it as a distinct edge kind (loop-carried / initial-empty) EXCLUDED from the intra-band topo sort, exactly as it already excludes the round-vec clear+reappend and the kInI-frozen reads. As written, V-LINEAR would reject every differential-SCC program whose frontiers feed back — the two artifacts included. Add an edge category (e.g. kFeedback) with its own validator (initial vec is empty-init; producer in phase P, consumer in phase P' of the NEXT epoch) rather than a plain RAW.

## [HIGH] COMMIT_SWEEP table-id order (G9/V-SWEEP-ORDER) has NO dependence edge forcing it, yet §4.6 asserts the PinnedOrder is a topological sort that V-LINEAR certifies. For an order-free op set, V-LINEAR is satisfied by ANY of the 4! permutations of the commit sweeps, so it cannot certify the specific byte-identity order (tc: 4,8,13,17; d5rn: 4,7,11,14,18,21,24). The dependence vocabulary does NOT derive this load-bearing ordering.

EVIDENCE: tc.program.ir:291-294 commit-sweep order is 4,8,13,17; d5rn.program.ir:401-407 is 4,7,11,14,18,21,24 — pure table-id order, NEITHER stratum nor SCC order (tc.drir §III S3, §IV: 'COMMIT order ... pinned INDEPENDENT of dep graph (no inter-sweep Vec/flag edge forces it)'). The sweeps share no Vec and write only their OWN table's kInI, so there is no WAR/WAW/RAW between any two sweeps. §4.6 says the linearization is 'a topological sort of DepGraph'; §5 lists V-SWEEP-ORDER as a validator but §4.6 gives it no teeth — a topo-sort check is order-blind for an edgeless subgraph. This is the artifact spec-gap #5 (COMMIT_SWEEP order under-constrained), confirmed against both IRs.

RECOMMENDATION: v3 must make table-id order LOAD-BEARING DATA that V-SWEEP-ORDER asserts DIRECTLY (compare pinned sweep positions to sorted table ids), NOT a consequence of V-LINEAR. Either synthesize WAW-ordering edges among sweeps (artificial, hides that they are genuinely independent) or, cleaner, promote V-SWEEP-ORDER to a standalone pin-checker over the sweep subsequence. §4.6's 'checked linearization is a topo sort' is a necessary but NOT sufficient certificate for order-free op sets — the spec must say so.

## [MEDIUM] retire-after-fires (§4.2) is derived from a UNION-over-arms effect set on FIXPOINT_FIRE, but only ONE arm actually reads kDelNow/kAddNow. The unioned effect set over-approximates and cannot say WHICH arm creates the retire hazard, so the '§4.2 WAR is DERIVED from effect sets' claim is weaker than stated for multi-arm ops — the dependence graph loses precision (conservatively correct, but not the exact derivation §4 advertises).

EVIDENCE: Verified in both IRs: tc.program.ir arm0 (ir:160) reads alive-at-claim {kInI,kDel,kDelNow}; arm1 (ir:169) reads survives-so-far {kInI,kDel} — NO kDelNow. Same asymmetry on the INSERT side: ir:237 in-new-sans-frontier {kAddNow} vs ir:246 in-new-with-frontier {no kAddNow}. tc.drir §IV explicitly records 'S2.D5→S2.D8 NO kDelNow WAR' and 'S2.A5→S2.A8 NO kAddNow WAR'. But §2.1's FIXPOINT_FIRE carries ONE effect set 'union over arms', which includes kDelNow — so the derived WAR fires for the whole op, not the specific arm. The precise graph fact (arm0-only) is stated in the artifact but is NOT derivable from the §2.1 op-level effect set as specified.

RECOMMENDATION: Either (a) declare effect sets PER-ARM (PlanTree-level) so the WAR derivation is exact, or (b) explicitly document in §4.2 that RETIRE-AFTER is derived from the op-level UNION (conservative superset) and that the arm-precise attribution is intentionally dropped. The current text ('DERIVED as WAR/RAW/WAW over the effect sets') overstates precision for multi-arm ops. A V-FIRE-ARMS validator asserting |arms|==count of same-SCC delta positions is also missing (artifact spec-gap #1).

## [MEDIUM] §1's 'single def edge = identity' contradicts the multi-def accumulating-queue vecs V53/V57 (and their d5rn analogs $add_queue:84, $delete_queue:75), which take a SEED def in S1/seed-band AND intra-round fold defs in S2 WITHOUT a clear between. The round-vec excuse ('cleared then re-appended each round') does not cover them — they accumulate across the stratum/phase boundary. The def/use model under-specifies this category, so 'def(Vec) = the single op that PRODUCES the vec's contents this band' is false for the queue vecs the ordering (seed-before-drain, back-edge) depends on.

EVIDENCE: tc.drir §II: V53 has TWO def edges (S1.5 SEED_FOLD append ir:119 + S2.D7 CHAIN_FOLD append ir:181); V57 has THREE (S1.6 seed ir:124 + S2.R1 REDERIVE ir:193 + S2.A7 CHAIN_FOLD ir:258). In tc.program.ir, $delete_queue:53 is appended at ir:119 (seed) and ir:181 (back-edge fold) and drained/cleared at ir:132/139 (per round) — it is NOT cleared between the seed append and the first claim drain. §1 def(Vec) says 'the single op that PRODUCES the vec's contents this band', excusing only round vecs (Δ_D/Δ_A) as clear+reappend. V53/V57 are neither single-def nor clear+reappend round vecs. The seed-before-drain RAW (§4.1) and the tc back-edge RAW (§4.5) both hang on these vecs, so getting their def-edge category wrong corrupts the derivation of two named orderings.

RECOMMENDATION: Add an explicit accumulating-queue Vec category (multi-append, no intervening clear, drained once per round) distinct from the round vec and the single-def vec. §1's def/use identity rule must permit multiple def edges for this category, and V-LINEAR must treat each append→drain pair as its own RAW (which the artifact's §IV already does implicitly). Artifact spec-gap #4 confirmed.

## [MEDIUM] Vec element_shape (values | ids | id+cols) has NO witness in the emitted IR — every vector is typed only by column signature (<u64> vs <u64,u64>). The shape is a constructor-attached semantic annotation with no IR ground truth, so V-OLD-EQUIV (the R1 byte-identity gate) CANNOT catch a shape misassignment: a wrong shape lowers to the same <u64...> vector-define. A shape-consistency validator against the def op's role is needed but unspecified in §5.

EVIDENCE: tc.drir §II labels V59 claimed-del-frontier as 'ids' and V66 fire-output as 'id+cols', but tc.program.ir declares $claimed_del:59<u64,u64> and $delete_queue:66<u64,u64,u64> — both plain <u64...> vectors with no shape marker. §1 (F-9) makes element_shape 'intrinsic, set at def', and §7.3 makes R1 a byte-identical-golden gate certified by V-OLD-EQUIV. Since the old discovery web has no ElementShape concept, V-OLD-EQUIV compares nothing about shape; and the emitted IR has no shape token, so byte-identity is blind to it. The shape is load-bearing for the future D5/WCOJ seek lowering (id-keyed vs value tuples) but nothing derives or checks it in R1.

RECOMMENDATION: Add a shape-consistency validator (e.g. V-SHAPE) asserting each Vec's element_shape is consistent with its producing op's role: CLAIM_DRAIN/overdelete/frontier defs ⇒ ids; FIXPOINT_FIRE fold-output defs carrying projected cols ⇒ id+cols; SEED_FOLD value frontiers ⇒ values. This is derivable from the def op's effect+role and must be a §5 always-on check, since the byte-identity gate cannot back-stop it. Artifact spec-gap #3 confirmed.

## [LOW] Monotone-side gate ELISION has no op-attribute representation, so the spec's elision hook is untestable and a future monotone-elision pass has no attribute to key on. The Access.pred is just kInNew whether the side is monotone (trivially-true, elidable) or differential (load-bearing). tc (all-differential) has zero elision sites and cannot exercise it; d5rn HAS the sites but the vocabulary carries no marker distinguishing an elidable monotone gate from a load-bearing differential one.

EVIDENCE: d5rn.program.ir:251 'check-member in-new {@X:108,@Y:109} in %table:7' — table:7 is edge, MONOTONE (d5rn.drir §0: T7 edge MONOTONE), so this InNew is trivially true and elidable; d5rn.drir:342-345 confirms 'edge is monotone so its InNew is trivially true, but the gate is emitted uniformly'. Contrast tc.program.ir:160 alive-at-claim in table:4 (differential T_tc) — load-bearing. Both lower to a bare check-member with pred=kInNew/kAliveAtClaim; the ACCESS op (§2.1) carries pred + context but NO 'gate-elidable-because-monotone' flag. The DRTableImpl.differential bit exists (§7.1) but ACCESS does not reference the accessed table's monotone-ness as a first-class elision attribute. This is a rewrite-substrate gap, not a correctness ordering — hence LOW.

RECOMMENDATION: Since DRTableImpl already carries the differential flag, the elision predicate is DERIVABLE at pass time (Access.table.differential==false && pred==kInNew ⇒ elidable). Document this in §3/§7(c) as the keying rule so R2's monotone-elision pass has a defined hook, OR add an explicit Access.monotone_trivial attribute. Not blocking for R1 (no elision in R1), but the vocabulary should state the derivation so the substrate claim (F-1 optimizability) holds. Artifact spec-gap #6 confirmed.
