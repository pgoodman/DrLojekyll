# Identity-lowering feasibility review — tc.program.ir vs v3 objects + declared lowering defaults

## Verdict

**FEASIBLE WITH CONDITIONS.** Walking `^flow:198` top to bottom, roughly 95% of the region tree maps cleanly to (v3 object, lowering default): every claim drain, frontier filter, seed fold, fixpoint fire, chain fold, rederive, retire, deferred filter, and commit sweep — including its full guard-nesting spine (G11), its sort-unique placement (G6), and its dead net_* emission (G8) — is reproducible from the §2.1 effect sets, §3 plan trees, and §A-2 pinned linearization with the four F-4-demoted lowering defaults. But the walk surfaces **two region instances backed by NO v3 object and NO declared default** (the mid-round input-queue `vector-clear`s and the `@22 += 1` epoch bump), and a cluster of undeclared tie-break/template defaults (round-body band order beyond retire, dual-append intra-`seq` order, id/name allocation, proc-frame shell) that byte-identity for family #1 (F-4) strictly requires. None of these break the v3 *factoring* — every gap is either a missing effect entry or a missing lowering-default declaration, not a missing concept — so the fix is spec amendments, not redesign. On §7: **construct-alongside-and-validate is correct and should be built first**; replace-discovery in R1 would force a same-diff emission rewrite, violating F-2 and the §9 risk-#5 mitigation.

## Findings (ranked)

**F1 [HIGH] — Mid-round input-queue clears have no v3 object and no lowering default.**
What: `vector-clear $delete_queue:53` (ir:139), `:66` (ir:147), `:72` (ir:155) and the INSERT mirrors `$add_queue:57/:111/:116` (ir:216, 224, 232) — each accumulating queue is cleared immediately after its claim drain so this round's fire appends don't replay next round. Evidence: CLAIM_DRAIN's §2.1 effect set lists drain + gate + append(s) only — **no `vector:clear` of the drained input queue**; §1's clear+reappend excuse covers only the Δ round vecs (V59-61/V121-123, ir:129-131/206-208); none of the four F-4 demotions (G11/G6/G9/G8) covers it. This clear is semantically load-bearing (without it the fixpoint never quiesces — drained rows re-claim forever), so it is arguably IR content, not even a lowering default. Recommendation: add `vector:clear(input-queue)` to CLAIM_DRAIN's in-round effect set (making single-vs-in-round distinguishable by clear as well as dual-append), or define a clear-after-drain discipline for accumulating-queue vecs in §1 alongside A-4, plus a validator (every in-round-drained accumulating queue is cleared before round end).

**F2 [HIGH] — The epoch bump `@22 += 1` (ir:96) is a dangling pinned-order entry with no object behind it.**
What: the artifact's §V linearization lists "epoch-bump @22+=1 (ir:96)" but §III has no op node for it, §2's five effect sub-domains have no global-scalar RMW, and `global u64 @22` (ir:7) has no value-node family (it is neither Vec, Table, nor StateCell). R2's generic lowering driven "ONLY by the v3 objects" cannot emit ir:96. Recommendation: either add an EPOCH_BUMP op with a `global:rmw` effect (or fold it into ProcFrame as declared prologue material with a lowering default: "flow-proc body opens with the epoch bump"), and give the global a value-node home — StateCell is the natural fit if its R3 reservation is relaxed to "engine scalars".

**F3 [MEDIUM] — Round-body and output-band internal order is only partially edge-forced; the residual tie-breaks are undeclared.**
What: within the OVERDELETE body, D1<D7 and D2<D4/D5 and D3<D6 are forced (A-1's drain-before-refill WAR on V53/V66/V72), and retire-at-tail is §4.2 — but (a) claim-drain order D1,D2,D3 (pinned, per artifact §V "no Vec edge forces D1 before D2 in round 0"), (b) fires-before-chain-folds (D4/D5 vs D6: no shared resource — D4 touches T_tc/T_join/V59/V66, D6 touches V60/T_proj/V72), (c) D6-before-D7 (keyed by *source* claimed-vec table-id 8<17, though their *target* table-ids order the other way, 17>4 — so WHICH table keys the tie-break is ambiguous), (d) S1.1-before-S1.2 (− before + sign order), (e) INSERT.output F1-F3 then F4-F6 (sign-major then table-id, ir:267-290), and (f) the `fixpoint-loop testing` vec list order V59,V60,V61 (ir:127) are all A-2 "lowering-default tie-breaks" that A-2 declares only for the commit band. Evidence: ir:132-155 (claim order), ir:156-181 (fires then folds), ir:174 vs ir:178, ir:97 vs ir:102. Recommendation: extend A-2 into a declared ROUND-BODY BAND TEMPLATE — bands in fixed order (claims, fires, chain-folds, retires), table-id within band **keyed by the drained source vec's debug table**, sign − before + where both signs share a band, output bands sign-major-then-table-id — and generalize V-SWEEP-ORDER into a per-band lowering-conformance check.

**F4 [MEDIUM] — A-1's loop-carried classification is stated per-EPOCH but tc needs it per-ROUND.**
What: the round-body dep graph is cyclic at round granularity too: S2.D7→S2.D1 RAW on V53 and S2.D4/D5→S2.D2 RAW on V66 are next-*round* edges (artifact §IV labels them "[feedback]"/"NEXT round"), exactly the cyclicity A-1 resolved for epochs. §4.6 says region-internal orders "self-certify," but without extending A-1's WAR-plus-loop-carried treatment to the FixpointRound's internal linearization, the region's own V-LINEAR has no topo sort to certify. Evidence: ir:181 (D7 appends V53) vs ir:133 (D1 drains V53); ir:164/173 vs ir:141. Recommendation: one-line amendment — A-1 applies at every region nesting level (epoch AND fixpoint round); V-LOOP checks drain-before-refill per round for round-carried vecs.

**F5 [LOW] — Dual-append order inside `if-claimed seq` is undeclared.**
What: all six dual-append sites emit persistent-set append before frontier append (ir:137/138, 145/146, 153/154, 214/215, 222/223, 230/231), but §2.1/G10 gives CLAIM_DRAIN two *unordered* append effect entries and A-4 says appends are commutative. Byte-identity needs the order pinned. Recommendation: declare "persistent-set before round-frontier" as a G11-family template default.

**F6 [LOW, UNVERIFIED] — ProcFrame shell material and id/name allocation are not covered by any declared default.**
What I confirmed: the `vector-define` prologue (ir:67-94, vec-id-ascending), `return-true` (ir:295), and the `^entry` band shell (`par` at ir:46, param clears ir:55-56, `call ^flow`, `return-false`), plus `^init:3`/`^receive` synthesis (ir:36-64), have no §2 op or declared default — ProcFrame is named in §0 but never specified. What I could NOT finish verifying: whether byte-identity additionally requires the lowering to reproduce the emitter's numeric id stream (`$delete_queue:53`, `@var:64`, etc.) — table/index ids come from dataflow (unchanged by R1), but vec/var ids look emission-order-allocated; if so, "ids allocated sequentially at first touch in pinned order" must be declared a lowering default or family #1's byte-identity gate silently depends on incidental allocator behavior. Recommendation: specify ProcFrame (prologue = vec defs in id order; epilogue = return token per proc kind; entry/receive band templates) and pin the id-allocation rule; verify the var-id stream claim against Stratum.cpp before R2.

**F7 [INFO] — Prior-judge findings are genuinely resolved for this artifact.** A-2 resolves commit-band order (F-4 demotion G9 makes it R2 data + V-SWEEP-ORDER conformance); A-3 resolves the arm-precision WAR (artifact §IV correctly records S2.D5→D8 as *no* kDelNow WAR); A-4 resolves V53/V57 multi-def; G6-as-per-edge-attribute exactly reproduces all 10 `vector-unique` sites (ir:97,102,115,120,132,140,148,209,217,225 — verified byte-level). The four fire PlanTrees carry (index-id, pivot-col) so the G3 self-join disambiguation lowers unambiguously (ir:157/166/234/243).

## §7 R1-integration assessment

**Construct-alongside-and-validate is the right first build, and the suite risk is low; replace-discovery is not viable in R1.** The emission half (Stratum.cpp:1936-2327) consumes the old discovery's shared state; replacing discovery would force a same-diff emission rewrite — exactly the F-2 violation and §9 risk-#5 "two-webs-alive stall" the spec's justification names. Alongside-mode is zero-emission-change, so all 155 goldens and the 4-mode suite are trivially preserved; the only new failure mode is a false-positive V-OLD-EQUIV abort, which is a compiler assert (loud, exit-134-visible), not a silent golden drift. The real risk is the *weakness* of the oracle, per the spec's own carried open question: the old scheduling fixpoint's only observable is integer strata, while DR-IR pins a linearization — comparing them is apples-to-oranges unless R1 **seeds the pinned order from the integer lift** (the open question's weaker option) rather than deriving it independently. Do that first: it makes V-OLD-EQUIV a real isomorphism check for family #1, de-risks R2, and the independent deriver can land later behind the same validator. Also close F1/F2 (spec amendments) before R1 code, since BuildDRFlow must construct those objects to be old-web-isomorphic.
