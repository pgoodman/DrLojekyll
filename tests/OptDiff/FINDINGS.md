### F15 [FIXED]: condition flip fails to rip back downstream rows when the gated data path is deletion-capable
- Found while pinning `cond_diff_flipflop`. Minimal divergent repro (min5): `#message support(i32 S) @differential. #message item(i32 X) @differential. #local backing(i32 S). #local stored(i32 X). #query gated(free i32 X). backing(S):support(S). stored(X):item(X). c:backing(_). gated(X):stored(X),c.` Driver: add item 1; add support 10 (gated: 1 in all modes); remove support 10 — gated must retract, but opt/nocf kept the stale row while nodf/none retracted (divergence keyed solely on -disable-dataflow-opt). Trigger: the tested data path must be deletion-capable (`item @differential`); with an add-only `item` all 4 modes agree. Both polarities symmetric: with a `!c` clause, opt also failed to retract `ungated` rows when the gate opened.
- Root cause, one site: `EvaluateConditionAndNotify` (lib/ControlFlow/Build/Build.cpp). The counter-flip rip-back region scans each user's table, and when the user `CanReceiveDeletions()` it gated each row's removal on the user's full top-down checker. That checker re-proves the row under current semantics — including the condition that just flipped against it — so it rejects every row and the removal arm never runs. Whether the checker tests the condition depends on dataflow opt: under opt the condition-guard TUPLE merges into the data view and the scanned view's checker calls the `^test:` proc (self-defeating); under -disable-dataflow-opt the scanned view keeps a data-only checker (retraction works), hence the mode split.
- All-modes-wrong variant (diffrun-invisible, F11-style; min6): with the condition and gated clause reading `@differential` messages directly (`c : support(_). gated(X) : item(X), c.`), every mode kept the stale row — the scanned view's checker there also bottoms out self-defeatingly. Verified wrong pre-fix in opt and none; fixed by the same change.
- Fix: the rip-back direction no longer calls the top-down checker at all — no correct row gate exists there, since the view's checker tests the very condition that just flipped. Removal regions now run for every scanned row and the per-successor state transitions (present -> unknown, then top-down recheck) do the filtering, which is exactly how the already-correct add-only shape worked (no checker gate when `!CanReceiveDeletions()`). The push-through (addition) direction keeps the checker: the condition now passes, so it degenerates to the intended data-validity re-proof.
- Verified: min5 and min6 pass all 4 modes with correct retraction; `cond_diff_flipflop` carries the full shape (`item @differential`; item retraction while gate open, while closed, and in a same-batch swap) and passes all 4 modes; targeted sweep cond_multi_setter, cond_in_induction{,_deep}, cond_both_polarities, cond_double_test, booleans{,_diff}, negate_3, optimize_5, deadflowelimination_4/5 all 4-mode green; ctest 3/3; full suite SUITE: PASS.

## Round 6 (checkpoint (c) slice-4 pre-implementation analysis, 2026-07-10)

### F17 [FIXED 2026-07-10, slice 4]: hoisted seeds break the phantom-pair propagation cancellation — stale claim-queue entries propagate a +1 with no compensating -1
- Found by hand-analysis of the emitted linear-tc IR against MD §5.0/§5.1.1 BEFORE any nonlinear code was written, then confirmed empirically; all 4 modes identically wrong AND the wrongness is latent across batches (invisible to golden diffing on any 2-batch case: the drifted state is counter-only until a later batch queries it). Neither diffrun nor the oracle-vs-final-state comparison of the standing 2-batch fixtures could see it.
- Repro (now standing case `tc_phantom_claim`): linear `t : t(F,X), edge(X,T)`; seed edges {(1,2),(1,4),(4,5)}; batch 2 is exactly MD §5.1.1's {-edge(1,2), +edge(2,5)}; batch 3 removes edge(4,5). Truth after batch 3: t = {(1,4),(2,5)}. Pre-fix: t(1,5) survives forever.
- Mechanism: the landed lowering hoists EVERY seed (both dual-join sections and both base-seed signs) ahead of the OVERDELETE loop, whereas MD §5.0 interleaves `-` seeds into OVERDELETE and `+` seeds into INSERT. MD's phase-monotonicity ("within INSERT only + folds") is what guarantees a queued crossing is still valid when its claim drains. Under the hoisted order, batch 2's phantom pair on join instance (2,1,5) [t(1,2) deleted x edge(2,5) added] fires the + seed FIRST: instance count 0->1 CROSSES and enqueues; OVERDELETE's fixpoint -1 brings it back to 0 but does not cross (kInI=0). Counts net exactly zero — but TryClaimAdd (which tested only kAdd) then claimed the dead instance, and the unconditional projection folded a phantom +1 into the witness count of t(1,5) (2 instead of 1). Batch 3's real retraction decrements 2->1: no zero crossing, no retraction. Del-side dual of the same class: a `-` crossing enqueued at seed and restored by a later same-batch `+` seed fold (two base rules, -e1(k)/+e2(k)) hit TryClaimDel's `assert(nr <= 0)` -> debug SIGABRT.
- Fix (include/drlojekyll/Runtime/Table.h): signed claim gates — TryClaimDel additionally requires `C_nr <= 0` at claim time (the assert demoted to a gate), TryClaimAdd additionally requires `Total > 0`. A stale entry whose crossing was canceled intra-batch is dropped; any later genuine crossing re-enqueues. Soundness traced (opus adversarial pass): C_nr is frozen >= 0 before OVERDELETE (every nonrecursive fold is a hoisted seed), so a REDERIVE row always has Total >= C_r > 0 at claim-add and is never wrongly dropped; the del gate only ever drops rows with genuine surviving nonrecursive support (the firewall's own condition). Companion fix: NetAdded gained a `!kInI` conjunct (frontier = genuine presence gain) — without it, a gate-dropped del claim's surviving kAdd on a presence-unchanged row leaked a spurious net-addition downstream (standing discriminator: `two_base_flip_downstream`).
- Bonus: the claim gates alone (before any nonlinear work) flipped `transitive_closure_diff2` and `cf16_3` green — their "cyclic self-support over-retention" was this bug, not a drain infidelity.
- Verified: tc_phantom_claim red pre-fix / green post-fix in all 4 modes; suite comm vs the 19-name HEAD baseline shows zero regressions; the MD §8 drain argument (firewall_cycle, nonlin_diag_selfloop) re-traced and green.

### F18 [FIXED 2026-07-10, checkpoint (d)]: sign-keyed negation gate is incompatible with every possible §5.4 crossover read — and its fixpoint-context variant read the wrong OVERDELETE predicate
- Found by hand-analysis of the negate read predicates against MD §5.1's seed schema BEFORE any crossover code was written (the slice-4 method), then independently re-proved by an adversarial reviewer; never observable at HEAD because the crossover terms did not exist yet (the miscounted pairings both fired zero times), so no repro predates the fix.
- The defect: the forward negation gate was keyed on the WALK'S SIGN (`+` walk gated on key absent-in-InNew, `−` walk on absent-in-InI — EmitChainStep, and `Present` on the eager path), but MD §5.1's schema reads are keyed on POSITION, sign-independent. Impossibility proof: for rule H :- P, !N, same-batch mixed cases (1) +P/+N and (2) +P/−N leave P's row-state IDENTICAL (added-this-batch: every membership predicate agrees), yet require opposite crossover firings; since the crossover's fold sign is already consumed by N's direction, no pred-scan predicate can distinguish them — so ANY crossover paired with the sign-keyed gate miscounts case (1) (lost derivation / negative counter at Commit) or case (3) (double decrement). Variant of the same defect: in fixpoint-refire context (a recursive rule stepping through a negate) the OVERDELETE-side gate read absent-in-InI where the fixpoint schema (oracle FixReads) demands absent-in-InNew for BOTH signs — latent in disassemble.dr's dead deletion path, zero golden coverage.
- Fix (with the (d) crossover, one atomic change): position-keyed, context-keyed gates — seed context reads key absent-in-InI for BOTH signs (Convention A, negated atom last; also makes the eager gate ingest-arm-order-independent, since InI is batch-frozen while Present is not); fixpoint-refire context reads absent-in-InNew for both signs; the eager gate reads InI (`@never` keeps `Present`); the crossover scans the predecessor at InNew. Monotone negated tables get a net-additions boundary vector + Seal enrollment in the same change (an InI gate on a never-sealed monotone table is vacuously false).
- Standing discriminators: negate_cobatch_diff (one @differential message feeding pred AND negated view — all four mixed cases in one genuine epoch, oracle-refereed), d5_recursive_negate (recursive rule through a negate over a differential lower view — exercises the fixpoint-context gate both signs), negate_lower_strata (negated view strictly below the pred — pins the runtime's global position convention against the oracle's per-rule order, both net-exact).
